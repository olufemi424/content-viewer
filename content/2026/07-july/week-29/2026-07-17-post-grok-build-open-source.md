## Hook
xAI just turned a coding-agent backlash into an open-source release.

That matters more than the model.

## Why this matters
If you are a creator, solo builder, or operator using coding agents on real repositories, the question is no longer just which model writes better code. It is whether you can inspect the runtime that touches your files. xAI announced on July 15, 2026 that Grok Build is now open source under Apache 2.0, including the CLI, the terminal UI, and the harness. The timing matters because it came right after public criticism over Grok Build sending repository data to xAI-controlled infrastructure. For builder audiences, that makes this a useful shift in buying criteria: trust, auditability, and local control are becoming product features.

## Mechanism
The change is concrete. xAI published the Grok Build codebase on GitHub and described it as a local-first coding agent that can run with different models and providers. That means the layer that plans tasks, runs commands, and handles repo context is no longer a black box. Builders can inspect the code, self-host pieces of the workflow, fork it, or at minimum review what the tool is doing before they put it near sensitive projects. In plain English: the competitive edge is moving from just model quality to whether the agent runtime itself can be trusted.

## Proof/use case
The best proof is the sequence of events. After developers raised alarms that Grok Build had uploaded code to xAI servers, xAI said it deleted retained data, turned retention off by default, and then open-sourced the agent runtime. That makes this more than a routine repo drop. It is a pressure-tested example of how fast coding-agent users will punish tools that feel opaque around source code handling. The practical use case is simple: if you run agents on client work, internal tools, or anything sensitive, start comparing products on three things now: what runs locally, what gets sent out, and what parts of the execution layer you can actually inspect.
